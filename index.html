Peer-to-peer mesh networking,
without the servers.

MyOwnMesh is a pure-Rust substrate for apps whose devices need to find and talk to each other directly. Devices on the same network ID discover each other through public Nostr relays, then talk over encrypted WebRTC — no central server you run, no port forwarding, no static IPs.

Embed it as a library, run it as a headless daemon, or drive it with the desktop GUI. Three personas, one substrate.

Install in one command

Detects your platform, fetches the matching binary from GitHub Releases, verifies the SHA-256, drops myownmesh on your PATH. 

# macOS / Linux
curl -fsSL https://raw.githubusercontent.com/mrjeeves/MyOwnMesh/main/scripts/install.sh | sh
# Windows (PowerShell)
irm https://raw.githubusercontent.com/mrjeeves/MyOwnMesh/main/scripts/install.ps1 | iex
# From source — needs cargo (https://rustup.rs)
git clone https://github.com/mrjeeves/MyOwnMesh
cd MyOwnMesh
cargo install --path crates/myownmesh         # myownmesh on $PATH via ~/.cargo/bin

Then open a new terminal and run myownmesh serve. Prefer a desktop UI? GUI bundles (.dmg · .msi · .exe · .deb · .AppImage) ship on GitHub Releases alongside the daemon and auto-spawn it as a child process — installing only the GUI gets you both.

What you build with it

Three shapes the substrate was designed for. The library treats payloads opaquely, so anything you can serialize fits.

case 1

An app that runs across every device you own

The motivating use case — MyOwnLLM's boxes on the same network ID find each other and redistribute work. A Raspberry Pi can borrow the workstation's model; the workstation can serve the laptop's chat. No account, no API key, no MyOwnLLM-operated broker.

case 2

Peer-to-peer inside your own Rust app

Add myownmesh-core as a git dependency, define your message types, and your app's devices start talking to each other in roughly fifteen lines of code. No GUI baggage, no HTTP self-updater, no Tauri — the library crate keeps the surface narrow.

case 3

A headless mesh on a fleet of Pis

The daemon is a single static binary that runs anywhere the matrix builds. linux-aarch64 covers Raspberry Pi 4 / 5. Manage the whole fleet from one desktop GUI, which talks to each daemon over its local control socket.

Embed in your Rust app

The library crates aren't on crates.io yet — pull them as git dependencies pinned to a release tag. Cargo dedupes by URL, so both crates resolve out of the same checkout. 

[dependencies]
myownmesh-core      = { git = "https://github.com/mrjeeves/MyOwnMesh", tag = "v0.1.2" }
myownmesh-signaling = { git = "https://github.com/mrjeeves/MyOwnMesh", tag = "v0.1.2" }  # Nostr driver
tokio = { version = "1", features = ["full"] }
use myownmesh_core::{Mesh, MeshConfig, NetworkConfig, TopologyMode};

#[tokio::main]
async fn main() -> Result<(), Box<dyn std::error::Error>> {
    let mesh = Mesh::open(MeshConfig::load().unwrap_or_default()).await?;

    let net = mesh.join(NetworkConfig {
        id: "home".into(),
        network_id: "my-cool-mesh".into(),
        label: "Home mesh".into(),
        topology: TopologyMode::default(),       // Ring
        signaling: Default::default(),            // Nostr defaults
        stun_servers: Default::default(),
        turn_servers: Default::default(),
        roster_path: None,
        auto_approve: false,
    }).await?;

    let _nostr = myownmesh_core::engine::attach_nostr(&net.state());

    let mut events = mesh.events();
    while let Ok(event) = events.recv().await {
        println!("{event:?}");
    }
    Ok(())
}

Set MYOWNMESH_HOME=~/.yourapp/mesh to keep your app's identity and rosters under its own directory tree (defaults to ~/.myownmesh/). Full walkthrough in docs/QUICKSTART.md — covers typed Channel<T>, generic Rpc, the approval workflow, and clean shutdown.

How it works

1

Nostr finds the peers

Each device subscribes to a room handle derived from SHA-256(app_id || ":" || network_id) on a deterministic shuffle of five public Nostr relays — same wire format as JS Trystero v0.24. No DNS-SD, no DHT, no central broker MyOwnMesh runs.

2

WebRTC carries the bytes

Once two peers find each other via the relay, they hand off to a DTLS-secured WebRTC data channel — direct if NAT lets them, STUN/TURN-assisted if it doesn't. Pure webrtc-rs; no libp2p, no custom UDP.

3

ed25519 confirms it's really them

First handshake exchanges nonces, each side signs the other's nonce under a domain-tag (so a v1 signature can't be replayed as v2), and a 6-char [a-z0-9] code rides along for eyeball confirmation. Approved peers persist in a per-network roster and skip the prompt next time.

4

A seven-tier ladder keeps it alive

Steady → Wake probe → ICE watchdog → ICE restart → Re-handshake → Room rejoin → Stop-and-start. Each tier is the cheapest action that recovers from the failure class above it. Every tunable is in CONNECTION-ENGINE.md with the field bug it was discovered through.

The daemon, day-to-day

Once myownmesh is on your PATH, the CLI drives everything else. 

myownmesh                  # start the daemon (alias for `serve`)
myownmesh serve            # run the daemon in foreground
myownmesh identity show    # print this device's id
myownmesh ctl status       # query a running daemon
myownmesh ctl networks list
myownmesh update check     # poll the release feed
myownmesh config edit      # open ~/.myownmesh/config.json in $EDITOR

The daemon reads ~/.myownmesh/config.json, joins every network listed there, attaches the Nostr signaling driver per network, and listens for myownmesh ctl … clients on a local socket (~/.myownmesh/daemon.sock on Unix, named pipe on Windows). Crashing the GUI never disturbs the running mesh.

One workspace, three personas

bin
myownmesh
Headless daemon + CLI. The thing the installer drops on your PATH.
crates/myownmesh
lib
myownmesh-core
The runtime, engine, and protocol. What you depend on to embed the mesh.
crates/myownmesh-core
app
myownmesh-gui
Desktop GUI — Tauri + Svelte 5. A client of the daemon over its local control socket.
gui/

Plus two supporting library crates the daemon and embedders share: myownmesh-signaling (Nostr driver + in-process LocalBroker for tests) and myownmesh-updater (self-update against a configurable release feed). An app embedding myownmesh-core pulls neither.
