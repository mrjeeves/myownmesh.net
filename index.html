A private mesh network
you actually own.

MyOwnMesh lets your devices find each other and talk directly — encrypted end to end, with no account, no port forwarding, and no static IPs. Put the same network ID on two machines and they connect.

And there's nothing underneath it you don't control. Discovery, NAT traversal, relaying — every piece of the plumbing is the same program. Run it on your laptop, across a tailnet, or on your own domain. It works out of the box, and it's exactly as secure and robust when you point it at infrastructure you run.

Get it

The desktop app is the easiest way in — a native bundle with the daemon built in, so a bare myownmesh just opens the app. Prefer the terminal? One command installs both.

Desktop downloads (.dmg · .msi · .deb · .AppImage) 
# macOS / Linux
curl -fsSL https://myownmesh.net/install.sh | sh
# Windows (PowerShell)
irm https://myownmesh.net/install.ps1 | iex
# From source — needs cargo (https://rustup.rs)
git clone https://github.com/mrjeeves/MyOwnMesh
cd MyOwnMesh
cargo install --path crates/myownmesh         # myownmesh on $PATH via ~/.cargo/bin

The script detects your platform, pulls the matching binaries from GitHub Releases, verifies the SHA-256, and installs both the myownmesh daemon and the desktop GUI — so a bare myownmesh opens the app (--no-gui / -NoGui for a headless, daemon-only box). Then open a new terminal and run myownmesh serve. The downloads page has the native installers, the portable daemon binary, and checksums.

What you build with it

Three shapes the substrate was designed for. The library treats payloads opaquely, so anything you can serialize fits.

case 1

One app across every device you own

Your phone, laptop, workstation, and a Pi in the closet — all on the same network ID, all finding each other and sharing work or state directly. No account to sign into, no API key, no broker in the middle. The devices are the service.

case 2

Peer-to-peer inside your own Rust app

Add myownmesh-core as a git dependency, define your message types, and your app's devices start talking to each other in roughly fifteen lines of code. No GUI baggage, no HTTP self-updater, no Tauri — the library crate keeps the surface narrow.

case 3

A headless mesh on a fleet of Pis

The daemon is a single static binary that runs anywhere the matrix builds. linux-aarch64 covers Raspberry Pi 4 / 5. Manage the whole fleet from one desktop GUI, which talks to each daemon over its local control socket.

Run the whole thing yourself

A fresh install connects on the project's reference servers, so it just works. But none of that is infrastructure you're stuck depending on — every piece is the same myownmesh binary, ready to host. Flip it on locally, behind a tailnet, or on your own domain, and point your peers at it.

①

Be your own discovery

myownmesh serve with the signaling role on is a rendezvous relay your peers connect to in place of the public default. Want it reachable over wss:// from anywhere? One command stands up a TLS reverse proxy that auto-provisions a certificate: myownmesh install caddy your-domain.

②

Be your own NAT traversal

Turn on the TURN role and you've got STUN and TURN on one port for the peers that can't connect directly (symmetric NAT, CGNAT, locked-down hotspots). STUN is answered by the TURN listener automatically — enable both and it just runs the one, nothing to wire up or trip over.

③

Run it anywhere — it's the same mesh

On a single LAN, riding on top of a tailnet, or out on a public domain: peers still find each other and connect the same encrypted way, directly when the network allows and relayed when it doesn't. Same security, same robustness, wherever you put it.

④

The fleet finds your servers

A host advertises the roles it offers, so the rest of the mesh discovers your relay, signaling, and STUN/TURN on its own — no per-device URLs to hand out. Toggle any role from the GUI, the CLI, or config.json. Enough to run a network that touches no infrastructure you don't own.

How it works

1

The room finds the peers

Each device subscribes to a room handle derived from SHA-256(app_id || ":" || network_id) on a signaling relay — the project's by default, or yours. No DNS-SD, no DHT, no account, no broker you have to trust with anything but a rendezvous.

2

WebRTC carries the bytes

Once two peers find each other, they hand off to a DTLS-secured WebRTC data channel — direct if NAT lets them, STUN/TURN-assisted if it doesn't. The relay never sees your traffic. Pure webrtc-rs; no libp2p, no custom UDP.

3

ed25519 confirms it's really them

The first handshake exchanges nonces, each side signs the other's under a domain-tag, and a short [a-z0-9] code rides along for eyeball confirmation. Approved peers persist in a per-network roster and skip the prompt next time. The signatures are what establish trust — not the relay.

4

It heals itself — fast

A seven-tier ladder takes the cheapest action that recovers each failure class, from a wake probe to an ICE restart to a full room rejoin. Close the lid and reopen it and the mesh notices the resume, redials, and has you back among your peers in seconds. Every tunable is in CONNECTION-ENGINE.md.

Embed in your Rust app

The library crates aren't on crates.io yet — pull them as git dependencies pinned to a release tag. Cargo dedupes by URL, so both crates resolve out of the same checkout. 

[dependencies]
myownmesh-core      = { git = "https://github.com/mrjeeves/MyOwnMesh", tag = "v0.1.10" }
myownmesh-signaling = { git = "https://github.com/mrjeeves/MyOwnMesh", tag = "v0.1.10" }  # Nostr driver
tokio = { version = "1", features = ["full"] }
use myownmesh_core::{Mesh, MeshConfig, NetworkConfig, TopologyMode};

#[tokio::main]
async fn main() -> Result<(), Box<dyn std::error::Error>> {
    let mesh = Mesh::open(MeshConfig::load().unwrap_or_default()).await?;

    let net = mesh.join(NetworkConfig {
        id: "home".into(),
        network_id: "my-cool-mesh".into(),
        label: "Home mesh".into(),
        topology: TopologyMode::default(),       // Ring
        signaling: Default::default(),            // reference relay; or point at your own
        stun_servers: Default::default(),
        turn_servers: Default::default(),
        roster_path: None,
        auto_approve: false,
    }).await?;

    let _nostr = myownmesh_core::engine::attach_nostr(&net.state());

    let mut events = mesh.events();
    while let Ok(event) = events.recv().await {
        println!("{event:?}");
    }
    Ok(())
}

Set MYOWNMESH_HOME=~/.yourapp/mesh to keep your app's identity and rosters under its own directory tree (defaults to ~/.myownmesh/). Full walkthrough in docs/QUICKSTART.md — covers typed Channel<T>, generic Rpc, the approval workflow, and clean shutdown.

The daemon, day-to-day

Once myownmesh is on your PATH, the CLI drives everything else. 

myownmesh                  # start the daemon (alias for `serve`)
myownmesh serve            # run the daemon in foreground
myownmesh service install  # run at boot/login (systemd · launchd)
myownmesh ctl status       # query a running daemon
myownmesh ctl services status   # relay / signaling / STUN / TURN, live
myownmesh install caddy my.domain   # TLS in front of your relay, one shot
myownmesh config edit      # open ~/.myownmesh/config.json in $EDITOR

The daemon reads ~/.myownmesh/config.json, joins every network listed there, attaches the signaling driver per network, and listens for myownmesh ctl … clients on a local socket (~/.myownmesh/daemon.sock on Unix, a named pipe on Windows). Crashing the GUI never disturbs the running mesh.

Where it's at

Version
0.1.10 — alpha. Wire format and API may still move; pin git tags.
Library on crates.io
Not yet. Use git dependencies pinned to a release tag.
Run your own, end to end
A device can host every piece of the plumbing itself — relay routing, a signaling relay (front it with myownmesh install caddy), and STUN / TURN. Toggle each from the GUI, the CLI, or config.json; hosts advertise their roles so the fleet self-discovers them. Enough to run a network on no infrastructure you don't own. See docs/SERVICES.md.
Network governance
Both open (default — any member writes to the roster) and closed (Member / Controller / Owner roles, authority enforced on the wire) ship today. Lifecycle in docs/NETWORK-TYPES.md.
Resilience
The seven-tier reconnection ladder, the verification-code approval UX, and the per-network roster model were all hardened through real production use before being generalised into the library.
What's next
More automatic relay, ring, and network formation — so a mesh spread across LANs, tailnets, and domains stitches itself together with even less to configure.
